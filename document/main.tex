\chapter*{Resumen}\addcontentsline{toc}{chapter}{Resumen}

Este informe describe el desarrollo de una aplicación web interactiva para el análisis de la mortalidad no fetal en Colombia durante el año 2019. La aplicación fue construida con \textbf{Python}, usando \textbf{Plotly} para la generación de gráficos y \textbf{Dash} como framework web, e integra siete visualizaciones sobre un conjunto de datos de 244\,226 registros oficiales del DANE.

El documento presenta la arquitectura de la solución, el procesamiento de los datos, la descripción de cada visualización implementada y la configuración de despliegue en la plataforma \textbf{Render}.

\textit{Palabras clave}: mortalidad, Colombia 2019, dashboard interactivo, Plotly, Dash, DANE, análisis de datos.

%% ─── CAPÍTULO 1: INTRODUCCIÓN ─────────────────────────────────────────────
\chapter{Introducción}\label{chap:introduccion}

\section{Contextualización}

En Colombia, el análisis de la mortalidad constituye una herramienta fundamental para la toma de decisiones en salud pública. El Departamento Administrativo Nacional de Estadística (DANE) publica anualmente los registros de defunciones no fetales, permitiendo identificar causas de muerte, distribución geográfica y patrones demográficos.

Esta actividad propone construir una aplicación web dinámica que facilite la exploración visual de dichos datos para el año 2019, haciendo accesible el análisis a través de una interfaz interactiva publicada en la nube.

\section{Objetivo General}

Desarrollar e implementar un dashboard interactivo en Python con Plotly y Dash que visualice los datos de mortalidad no fetal de Colombia para el año 2019, incluyendo siete representaciones gráficas complementarias y desplegado en una plataforma en la nube.

\section{Objetivos Específicos}

\begin{enumerate}
    \item Procesar y limpiar los conjuntos de datos oficiales del DANE para su uso en el dashboard.
    \item Implementar las siete visualizaciones requeridas: mapa coroplético, gráfico de líneas, gráfico de barras, gráfico circular, tabla, barras apiladas e histograma.
    \item Estructurar el código en capas separadas (carga de datos, procesamiento y presentación) siguiendo principios de diseño limpio.
    \item Desplegar la aplicación en la plataforma Render como servicio web accesible públicamente.
\end{enumerate}

\section{Consideraciones en el preprocesamiento}

Durante la carga de datos se identificaron y resolvieron dos problemas específicos:

\begin{enumerate}
    \item \textbf{BOM en la primera columna}: el archivo de mortalidad incluía una marca de orden de bytes (\textit{Byte Order Mark}) al inicio. Se resolvió con \texttt{encoding="utf-8-sig"} al leer el CSV y renombrando la primera columna explícitamente.

    \item \textbf{Comas finales inconsistentes en el catálogo CIE-10}: el archivo \texttt{Anexo2} presentaba comas adicionales al final de cada fila, generando 7 campos en lugar de 6. La solución empleada fue un lector manual con \texttt{csv.reader} que trunca cada fila a los primeros 6 campos, descartando el campo espurio.
\end{enumerate}

\section{Estructura del proyecto}

La aplicación sigue una separación estricta de responsabilidades en tres capas: carga de datos, procesamiento analítico y presentación visual.

\begin{lstlisting}[language={}, caption={Estructura de archivos del proyecto}, label={lst:tree}]
analytics/
  app.py                  # Punto de entrada Dash / Gunicorn
  layout.py               # Ensamble del dashboard completo
  Procfile                # Configuracion para Render
  requirements.txt        # Dependencias Python
  data/
    loader.py             # Carga y cacheo de DataFrames
    processor.py          # Agregaciones analiticas (7 funciones)
    raw/                  # Archivos CSV y GeoJSON
  components/
    map_chart.py          # Mapa coro pletico por departamento
    line_chart.py         # Muertes por mes
    bar_chart.py          # 5 ciudades mas violentas
    pie_chart.py          # 10 ciudades con menor mortalidad
    stacked_bar.py        # Muertes por sexo y departamento
    histogram.py          # Distribucion por grupo de edad
\end{lstlisting}

\section{Capa de datos — \texttt{data/loader.py}}

El módulo \texttt{loader.py} expone cuatro funciones decoradas con \texttt{@lru\_cache(maxsize=1)}, garantizando que cada DataFrame se lea desde disco exactamente una vez durante el ciclo de vida de la aplicación. Esto es crítico en producción: con 244\,226 registros, la carga inicial toma aproximadamente 2.5 segundos; todas las solicitudes posteriores usan el caché en memoria.

La línea clave es \texttt{app.layout = build\_layout()} (no la función, sino su resultado): el layout se construye una vez en el inicio del proceso y se almacena en memoria. La variable \texttt{server = app.server} expone el servidor Flask subyacente de Dash, necesario para que Gunicorn pueda servir la aplicación.

%% ─── CAPÍTULO 4: VISUALIZACIONES ─────────────────────────────────────────
\chapter{Visualizaciones Implementadas}\label{chap:visualizaciones}

\section{Mapa coroplético — distribución de muertes por departamento}

Visualiza el total de fallecimientos en cada departamento de Colombia mediante un mapa coroplético con escala de color ocre. Los datos de mortalidad usan códigos DANE de 2 dígitos que deben rellenarse con cero a la izquierda (\texttt{zfill(2)}) para coincidir con la propiedad \texttt{DPTO} del GeoJSON. La clave de unión es \texttt{featureidkey="properties.DPTO"}.

\section{Gráfico de líneas — muertes por mes}

Representa la variación mensual de defunciones de enero a diciembre de 2019. Se agrupa por la columna \texttt{MES} y se visualiza con área rellena bajo la curva para evidenciar patrones estacionales.

\section{Gráfico de barras — 5 ciudades más violentas}

Identifica las cinco ciudades con mayor número de homicidios con arma de fuego, filtrando los registros cuyo código CIE-10 comienza con \texttt{X95} (agresión con disparo de arma de fuego, especificada y no especificada).

\section{Gráfico circular — 10 ciudades con menor mortalidad}

Identifica los diez municipios con el menor total de defunciones registradas, excluyendo aquellos con cero muertes (que indicarían ausencia de cobertura de datos, no ausencia real de mortalidad). Se presenta como donut con la paleta ocre definida en el proyecto.

\section{Tabla — 10 principales causas de muerte}

Cruza los registros de mortalidad con el catálogo CIE-10 para obtener el nombre de cada causa. Presenta las diez más frecuentes ordenadas de mayor a menor, con las columnas: código, nombre de la causa y total de casos. Se implementó con \texttt{dash\_table.DataTable} para permitir ordenamiento interactivo.

\section{Gráfico de barras apiladas — muertes por sexo y departamento}

Compara el total de defunciones por sexo (masculino, femenino e indeterminado) en cada departamento, permitiendo identificar diferencias regionales. Los valores de la columna \texttt{SEXO} se mapean: 1 → Masculino, 2 → Femenino, 3 → Indeterminado.

\chapter{Despliegue en Render}\label{chap:despliegue}

\section{Plataforma seleccionada}

La aplicación está preparada para despliegue en \textbf{Render} como \textit{Web Service}. Render fue elegido por ofrecer un plan gratuito funcional para proyectos académicos, integración directa con repositorios Git, detección automática del runtime Python y soporte nativo para Gunicorn.

\begin{table}[H]
    \centering
    \caption{Configuración del Web Service en Render}
    \label{tab:render}
    \begin{tabular}{ll}
        \toprule
        \textbf{Parámetro} & \textbf{Valor} \\
        \midrule
        Tipo de servicio  & Web Service \\
        Runtime           & Python 3 \\
        Build Command     & \texttt{pip install -r requirements.txt} \\
        Start Command     & \texttt{gunicorn app:server -{}-workers 1 -{}-threads 2 -{}-timeout 120} \\
        Plan              & Free (gratuito) \\
        \bottomrule
    \end{tabular}
\end{table}

\section{Archivos de configuración}

El archivo \texttt{Procfile} define el comando de inicio para Gunicorn:

\begin{lstlisting}[language={}, caption={Procfile para Render}, label={lst:procfile}]
web: gunicorn app:server --workers 1 --threads 2 --timeout 120
\end{lstlisting}

El parámetro \texttt{-{}-timeout 120} es necesario porque la primera solicitud carga los 244\,226 registros en memoria; un timeout de 30 segundos (valor por defecto de Gunicorn) sería insuficiente.

Las dependencias de producción se especifican en \texttt{requirements.txt}:

\begin{lstlisting}[language={}, caption={requirements.txt con versiones fijadas}, label={lst:req}]
dash==2.18.2
plotly==5.24.1
pandas==2.2.3
gunicorn==23.0.0
\end{lstlisting}

\section{Consideraciones del plan gratuito}

El plan gratuito de Render tiene la limitación de que el servicio entra en modo de suspensión tras 15 minutos de inactividad. Al recibir una nueva solicitud, el proceso debe reiniciarse, lo que implica recargar todos los datos y puede tardar entre 30 y 60 segundos. Para un uso académico de demostración, esta limitación es aceptable.

%% ─── CAPÍTULO 6: CONCLUSIONES ────────────────────────────────────────────
\chapter{Conclusiones}\label{chap:conclusiones}

La aplicación desarrollada cumple íntegramente los requisitos de la actividad: implementa las siete visualizaciones sobre los datos de mortalidad no fetal de Colombia 2019, organiza el código en capas claramente separadas y está lista para despliegue en la nube.

Los aprendizajes principales obtenidos durante el desarrollo incluyen:

\begin{itemize}
    \item \textbf{Plotly y Dash}: construcción de visualizaciones interactivas y ensamble de dashboards multi-panel con estilos personalizados.
    \item \textbf{Manejo de datos geoespaciales}: uso de GeoJSON con Plotly Choropleth y correspondencia de claves de unión entre DataFrames y propiedades del GeoJSON.
    \item \textbf{Optimización con caché}: el uso de \texttt{lru\_cache} fue determinante para hacer la aplicación viable en producción con datos de gran volumen.
    \item \textbf{Limpieza de datos}: el preprocesamiento de los archivos DANE requirió resolver problemas reales de calidad de datos (BOM, comas inconsistentes) que no serían evidentes sin análisis previo.
    \item \textbf{Despliegue en la nube}: configuración de Gunicorn como servidor WSGI y uso de Render como plataforma PaaS.
\end{itemize}
